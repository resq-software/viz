## Critical Issues (Must Fix)

### Issue 1: Clickable `<div>` cards have no keyboard support — **WCAG 2.1.1 Keyboard (A)**, **4.1.2 Name/Role/Value (A)**

**Severity:** Critical
**Location:** `src/ResQ.Viz.Web/client/index.html:140-152` (scenario-card × 4), `:162-178` (terrain-card × 5)
**Wired in:** `src/ResQ.Viz.Web/client/controls.ts:52` (scenario), `:78` (terrain)

**Problem:**
9 `<div class="scenario-card|terrain-card">` elements have JS `click` listeners but no `role="button"`, no `tabindex="0"`, and no `keydown` handler. Keyboard-only and screen-reader users cannot trigger them.

**Code Fix:**
```html
<!-- Before -->
<div class="scenario-card" data-scenario="single">
    <div class="sc-name">Single</div>
    <div class="sc-desc">1 drone</div>
</div>

<!-- After -->
<button class="scenario-card" type="button" data-scenario="single">
    <span class="sc-name">Single</span>
    <span class="sc-desc">1 drone</span>
</button>
```

Use `<button>` instead of `<div>` — it gets keyboard, focus, and role for free. Strip default button styles in CSS (`.scenario-card { all: unset; ... }` or reset `border/background/font`). Same fix for `.terrain-card`.

**Testing:** Tab to a card, press Enter and Space — both must trigger the same action as click. Screen reader should announce "Single, 1 drone, button".

### Issue 2: Form labels not associated with their inputs — **WCAG 1.3.1 Info and Relationships (A)**, **3.3.2 Labels or Instructions (A)**

**Severity:** Critical
**Location:** `src/ResQ.Viz.Web/client/index.html:204-214` (spawn X/Y/Z), `:229-232` (wind speed/dir)

**Problem:**
Labels are siblings, not parents, and have no `for=` attribute:
```html
<label class="field-label">X (East)</label>
<input type="number" id="spawn-x" value="0">
```
Screen readers cannot associate "X (East)" with the input. Clicking the label doesn't focus the input.

**Code Fix (two options):**
```html
<!-- Option A: add for= -->
<label class="field-label" for="spawn-x">X (East)</label>
<input type="number" id="spawn-x" value="0">

<!-- Option B: wrap (matches the settings panel pattern already used at index.html:340) -->
<label class="field-label">
    <span>X (East)</span>
    <input type="number" id="spawn-x" value="0">
</label>
```

Apply the same fix to `spawn-y`, `spawn-z`, `wind-speed`, `wind-dir`. The settings panel (`index.html:332-417`) already does this correctly — use it as the pattern.

**Testing:** Click each label; corresponding input must focus. NVDA/VoiceOver must read "X East, edit" when input is focused.

## Serious Issues

### Issue 3: Icon-only buttons use `title` instead of `aria-label` — **WCAG 4.1.2 Name/Role/Value (A)**

**Severity:** Serious
**Location:** `src/ResQ.Viz.Web/client/index.html:112-120` (HUD toggles), `:131-133` (Start/Stop/Reset have text — OK), `:296` (key-hints close)

**Problem:**
Buttons whose visible text is a unicode glyph (`⇒ ○ ⋯ ◎ ☰ ⚙ ✕`) rely on the `title` attribute for an accessible name. `title` is unreliable: not exposed by all screen readers, not surfaced on touch, hidden until hover.

**Code Fix:**
```html
<!-- Before -->
<button id="hud-vel-toggle" class="hud-toggle active" title="Velocity vectors (V)">⇒</button>

<!-- After -->
<button id="hud-vel-toggle" class="hud-toggle active"
        aria-label="Velocity vectors" title="Velocity vectors (V)">
    <span aria-hidden="true">⇒</span>
</button>
```

Apply to: `hud-vel-toggle`, `hud-halo-toggle`, `hud-form-toggle`, `hud-follow-toggle`, `btn-sidebar-toggle`, `hud-settings-toggle`, `key-hints-close`. `hud-hints-toggle` (`?`) reads OK as "question mark" but should also get an `aria-label="Keyboard shortcuts"`.

### Issue 4: Toggle buttons don't expose pressed state — **WCAG 4.1.2 Name/Role/Value (A)**

**Severity:** Serious
**Location:** `src/ResQ.Viz.Web/client/index.html:112-115`, JS in `app.ts:495`

**Problem:**
Four HUD toggles (velocity/halo/formation/follow) visibly toggle on/off via the `.active` CSS class. Only `hud-hints-toggle` has `aria-pressed="false"`, and `grep` finds **no JS code that updates `aria-pressed`** on any toggle. Screen-reader users never learn the toggle's current state.

**Code Fix:**
```html
<button id="hud-vel-toggle" class="hud-toggle active" aria-pressed="true" aria-label="Velocity vectors">…</button>
```

In `app.ts:495` (toggle click handler), every time you flip `.active`, mirror it:
```ts
btn.addEventListener('click', () => {
    const pressed = btn.classList.toggle('active');
    btn.setAttribute('aria-pressed', String(pressed));
    /* …existing toggle logic… */
});
```

### Issue 5: No `prefers-reduced-motion` support — **WCAG 2.3.3 Animation from Interactions (AAA, but expected for AA-quality)**

**Severity:** Serious (vestibular disorder accessibility)
**Location:** `src/ResQ.Viz.Web/client/styles/main.css` (32 `transition`/`animation` declarations, zero `prefers-reduced-motion` blocks)

**Problem:**
The viz has pulsing dots, wind compass animation, weather mode transitions, and HUD hover transitions. Users with vestibular disorders or who set OS-level reduced-motion get the full animation suite.

**Code Fix:** Append to `main.css`:
```css
@media (prefers-reduced-motion: reduce) {
    *, *::before, *::after {
        animation-duration: 0.01ms !important;
        animation-iteration-count: 1 !important;
        transition-duration: 0.01ms !important;
        scroll-behavior: auto !important;
    }
    /* Keep essential telemetry-rate animations subtle */
    .conn-dot.reconnecting { animation: none; opacity: 0.7; }
}
```

The 3D scene itself is exempt (it's the product), but UI chrome should respect the preference.

### Issue 6: No skip link — **WCAG 2.4.1 Bypass Blocks (A)**

**Severity:** Serious
**Location:** `src/ResQ.Viz.Web/client/index.html:55` (immediately after `<body>`)

**Problem:**
Keyboard users tabbing from the address bar must traverse the HUD bar (8+ buttons) and the entire sidebar (~30 controls) before reaching scenario cards. The only skip link present is inside the `<noscript>` block, which JS users never see.

**Code Fix:**
```html
<body>
    <a href="#scene-container" class="skip-link">Skip to 3D viewer</a>
    <a href="#sidebar" class="skip-link">Skip to controls</a>
    <main id="scene-container">…
```

CSS:
```css
.skip-link {
    position: absolute; left: -9999px; top: auto;
    padding: 8px 16px; background: var(--accent); color: #fff;
    z-index: 1000; border-radius: 0 0 4px 0;
}
.skip-link:focus { left: 0; top: 0; }
```

## Moderate Issues

### Issue 7: 3D canvas has no text equivalent for telemetry — **WCAG 1.1.1 Non-text Content (A)**

**Severity:** Moderate
**Location:** `src/ResQ.Viz.Web/client/index.html:58` (`<main id="scene-container">`)

**Problem:**
Screen-reader users get nothing from the 3D scene. The `<canvas>` is the entire product.

**Recommendation:** Add an `aria-live="polite"` text region (offscreen or visible) that summarises state when it changes — e.g., *"5 drones active. Lead drone at 23 m altitude, battery 78%. SAR scenario."*. The data already exists in `dronePanel.ts` and `hud.ts`. This won't make the viz fully accessible to blind users but it makes it auditable rather than opaque.

### Issue 8: Heading hierarchy invisible to JS users — **WCAG 1.3.1 Info and Relationships (A)**

**Severity:** Moderate
**Location:** `src/ResQ.Viz.Web/client/index.html:430` (only `<h1>` is inside `<noscript>`); `:380` jumps straight to `<h3>`

**Problem:**
The JS-rendered page has no `<h1>` and only `<h3>` headings inside the settings panel. Document outline is broken.

**Code Fix:** Add a visually-hidden `<h1>` inside `<main>`:
```html
<main id="scene-container">
    <h1 class="sr-only">ResQ Viz drone simulation viewer</h1>
    …
```
And rename the panel-section `.section-title` divs (e.g., "Simulation", "Scenarios", "Terrain") to `<h2>` semantically; the `<h3>` settings titles will then nest correctly.

### Issue 9: Custom card focus styles missing — **WCAG 2.4.7 Focus Visible (AA)**

**Severity:** Moderate
**Location:** `src/ResQ.Viz.Web/client/styles/main.css` (only 5 `:focus`/`:focus-visible` rules total, none for `.scenario-card` / `.terrain-card`)

**Problem:** Once Issue 1 makes cards keyboard-focusable, they need a visible focus ring or they become invisible to keyboard users.

**Code Fix:**
```css
.scenario-card:focus-visible,
.terrain-card:focus-visible {
    outline: 2px solid var(--accent);
    outline-offset: 2px;
    background: var(--accent-soft);
}
```
