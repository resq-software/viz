## Serious Issues

### Issue 1: `--text-faint` color contrast 2.77:1 — **WCAG 1.4.3 Contrast Minimum (AA)**

**Severity:** Serious — real WCAG AA failure, used 14× in CSS
**Location:** `src/ResQ.Viz.Web/client/styles/main.css:42` (token), then 14 call-sites including `:187, :224, :395, :429, :634, :671, :767, :904, :953, :972, :1316, :1362, :1533`

**Problem:**
Computed contrast of `#4f5866` (text-faint) on `#07090d` (bg) = **2.77:1**, well below the 4.5:1 minimum for normal text. Used across telemetry timestamps, panel separators, badge labels, event-log times, and section sub-headings.

**Measured ratios** (against `--bg #07090d`):
| Token | Hex | Ratio | AA pass? |
|---|---|---|---|
| `--text-faint` | #4f5866 | **2.77:1** | ❌ FAIL |
| `--text-muted` | #7a8593 | 5.32:1 | ✅ |
| `--text-dim` | #b3bcc8 | 10.38:1 | ✅ |
| `--text` | #e6edf3 | 16.86:1 | ✅ |

**Code Fix:** Lift the token until it crosses 4.5:1. `#7a8593` already passes — bump `--text-faint` to roughly `#6b7480` (5.0:1) which keeps the visual hierarchy below `--text-muted`.
```css
:root {
-  --text-faint:   #4f5866;
+  --text-faint:   #6b7480;  /* lifted from 2.77:1 → ~5.0:1 to meet WCAG AA 1.4.3 */
}
```
Single-line change, ripples through all 14 call-sites.

**Testing:** Open browser devtools → Lighthouse → Accessibility → confirm "Background and foreground colors do not have a sufficient contrast ratio" no longer fires for the `.el-time`, `.mc-sep`, panel-collapsible summaries, etc.

### Issue 2: Settings panel `aria-hidden="true"` never toggled — **WCAG 4.1.2 Name/Role/Value (A)**, **1.3.1 Info and Relationships (A)**

**Severity:** Serious — entire settings panel is invisible to assistive tech, even when visually open
**Location:** `src/ResQ.Viz.Web/client/index.html:330` (hardcoded `aria-hidden="true"`); JS that opens the panel: `app.ts:115-127`

**Problem:**
The `<div id="settings-panel" aria-hidden="true">` ships with `aria-hidden="true"` in HTML. JS adds/removes a CSS class to show it but **never updates `aria-hidden`**. Screen readers treat the panel as permanently hidden — keyboard users can tab into it but their AT announces nothing.

**Code Fix:**
1. Remove the hardcoded `aria-hidden="true"` from index.html, or
2. Better: wire AT visibility into the open/close handlers. In `app.ts` near the settings toggle:
```ts
function _setSettingsVisible(v: boolean): void {
    settingsPanel?.classList.toggle('open', v);
    settingsPanel?.setAttribute('aria-hidden', String(!v));
    settingsToggle?.setAttribute('aria-expanded', String(v));
}
```
Also add `aria-expanded` to `#hud-settings-toggle` in HTML (initial value `"false"`).

**Testing:** Open settings panel, then run NVDA → expect to hear "Settings dialog, Bloom, checkbox checked, …". Currently you hear nothing.

## Moderate Issues

### Issue 3: Active card state not exposed to AT — **WCAG 4.1.2 Name/Role/Value (A)**

**Severity:** Moderate
**Location:** `src/ResQ.Viz.Web/client/index.html:142-187` (cards), `app.ts:280, :333` (where `.active` toggles)

**Problem:**
Scenario and terrain cards mark "selected" with `.active` CSS only. Screen-reader users tabbing through the four scenario buttons hear `"Single, button"` × 4 with no indication of which is currently active.

**Code Fix:** Add `aria-pressed` to the cards (treat them as toggle buttons) or `aria-current="true"` (treat them as the current selection in a set). For a single-active-at-a-time pattern, `aria-pressed` is conventional:

```ts
// app.ts:278-281 — terrain
document.querySelectorAll<HTMLElement>('.terrain-card').forEach(el => {
    const active = el.dataset['preset'] === key;
    el.classList.toggle('active', active);
    el.setAttribute('aria-pressed', String(active));
});

// app.ts:333 — initial alpine selection
document.querySelector<HTMLElement>('.terrain-card[data-preset="alpine"]')
    ?.classList.add('active');
document.querySelector<HTMLElement>('.terrain-card[data-preset="alpine"]')
    ?.setAttribute('aria-pressed', 'true');
```
Initialise all unselected cards to `aria-pressed="false"` in HTML, or add an init pass in JS.

Apply the same pattern to scenario cards in `controls.ts:52` (the click handler that POSTs to `/api/sim/scenario/{name}`).

## Minor Issues

### Issue 4: Sliders have no human-readable value for AT — **WCAG 1.3.1 Info and Relationships (A)**

**Severity:** Minor
**Location:** Settings panel range sliders at `index.html:340, :353, :365, :373, :415` (5 sliders) and weather sliders `:230, :232`

**Problem:**
`<input type="range">` exposes raw min/max/value to AT, but the human-readable formatted value (e.g. `"0.55"`, `"60°"`, `"20%"`) lives in a sibling `<span class="settings-val">`. Screen readers say "Bloom strength, slider, 0.55" — fine for unitless values, but FOV reads as "Field of view, slider, 60" with no degree unit, and battery warn reads as "20" not "20 percent".

**Code Fix:** Either set `aria-valuetext` whenever the value updates in JS, or use `aria-describedby` to point at the existing val span:
```html
<input type="range" id="set-fov" min="40" max="100" step="5" value="60"
       aria-describedby="set-fov-val">
<span class="settings-val" id="set-fov-val">60°</span>
```
The browser already concatenates the description, so AT users hear "Field of view, slider, 60, 60 degrees".
